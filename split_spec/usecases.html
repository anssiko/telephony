This document defines the main use cases and requirements to the Telephony API.

Terminology

The terminology of the Telephony API is used.

Introduction

Stakeholders

The Web Telephony API stakeholders are the following:

  1. implementor: implements the API specification in a Web runtime, using the underlying platform features
  2. developer: implements applications using the API specification
  3. end user or user: provides input to the applications.

Use Cases

The main use cases of the Telephony API from the end users point of view are the following:

  1. make a call to a remote party
  2. id multiple subscriber identities are configured, choose which identity should be used for a dialed call
  3. set a default subscriber identity, so that the identity is optional to provide before dialing
  4. hide the caller identity, if the network supports it
  5. send DTMF tones to the telephony network
  6. list the usable emergency numbers
  7. dial an emergency call without having a SIM card
  8. receive phone calls
  9. put calls on hold
  10. resume held calls
  11. handle waiting calls, when the service is available and enabled
  12. deflect an incoming call to be handled by another remote party
  13. transfer an ongoing call with a remote party to become a call between the remote party and a third party
  14. add new call participants to a call or multiparty call
  15. remove call participants from a call
  16. see and manage call capabilities during a call, such as is multiparty call possible, is video possible, etc
  17. use headphones and external displays with calls
  18. select between audio sources (microphones) and audio targets (speaker, headphones, etc)
  19. set the default sound volume for telephony
  20. set the sound volume during a call
  21. mute and unmute the microphone for a call
  22. set and check network services, such as call forwarding, call barring, network selection mode, call counters, etc.
  23. manage the list of past calls through call history.

The main use cases from the developers point of view are to use the API to programmatically implement the user use cases from above. In addition,

  1. track the progress of calls through call state change event notifications
  2. track enabled and disabled telephony services through event notifications
  3. use a well defined structure for call history, for portability reasons

Requirements

The Telephony API MUST fulfill the following technical requirements:

  1. The API MUST be agnostic of implementation details such as underlying platform architecture and telephony middleware implementation.
